# Guidebook Data Schema

`guidebook-data.json` is generated by `scripts/build-guidebook-data.mjs` and rendered by `scripts/build-guidebook.mjs`. Keep it structured, compact, and focused on the facts needed by the stable guidebook UI; the renderer also reads sibling `itinerary-structured.json`, `map-data.json`, and Markdown source files when present.

Build traveler-facing content primarily from the three Markdown reports:

- `destination-brief.md`: overview, weather/season, arrival/departure/local transport, city-level warnings.
- `itinerary.md`: trip parameters, overall plan, every day timeline, daily meals/rest/budget/reservation/backup notes, transport/lodging references, budget, checklist.
- `reputation.md`: POI dossiers, avoid notes, hotel area strategy, accepted hotel candidates.

Use `itinerary-structured.json` as the lossless machine index of `itinerary.md`: it may copy table cells, raw Markdown rows, daily detail labels, transport links, budget rows, and hotel candidate rows, but it must not summarize or replace the Markdown report. Use `map-data.json` and `research-ledger.json` only to enrich those facts with coordinates, photos, route estimates, source IDs, freshness, or links. They must not introduce traveler-facing hotel names, attraction dossiers, daily plans, transport references, or budget rows that conflict with or replace the Markdown reports. `itinerary-data.json` is deprecated and ignored by guidebook generation if an old file is present.

## Required Base Fields

```json
{
  "destination": "成都",
  "dateRange": "2026-06-03 至 2026-06-06",
  "duration": "4天3夜",
  "travelers": "一家三口，含7岁小朋友",
  "origin": "广州",
  "generatedAt": "2026-05-09",
  "itinerarySource": "itinerary-structured.json",
  "sourceFactIds": ["fact_panda_base_duration_20260509"],
  "days": []
}
```

## Flexible Optional Modules

Only include modules when useful evidence exists. Do not force every guidebook to contain every module.

- `overview`: summary, highlights, warnings, pace, stayArea, transportMode, budgetLevel.
- `days`: daily themes, timeline activities, meals, rest, budget, booking reminders, backup notes.
- `poiExperiences`: attraction play time, why-go reasons, must-do projects, queue/reservation, preparation, family notes, common complaints, avoid notes, suitable/not-suitable fit, representative image when available, source.
- `transport`: city-to-city and local movement references. Flight/train rows may include `url`, `bookingUrl`, `booking_url`, `ticketUrl`, `ticket_url`, `detailUrl`, or `jumpUrl`; keep real FlyAI/Feizhu/airline/12306 links when returned so the guidebook can render "订票/查询".
- `map`: guidebook map display notes and optional viewport/category hints.
- `hotels`: hotel candidates, stay-area fit, tier/type tradeoffs, booking links, room-type status, and freshness notes.
- `food`: food items, child-friendly notes, and dining warnings. When a separate `food` block is not available, keep detailed meal rows in `itinerary.md` and food/avoid notes in `destination-brief.md` and `reputation.md`; the renderer can derive the "美食与口碑避雷" module from those source documents.
- `avoidance` or `reputationWarnings`: only meaningful warnings, not filler.
- `budget`: budget rows and total.
- `checklist`: grouped preparation items.
- `sourceFactIds`: ledger fact IDs that shaped the guidebook.

  "whyGo": ["亲子教育价值高", "早晨熊猫活跃度高"],
  "recommendedDuration": "3-4小时",
  "mustDo": ["月亮产房", "太阳产房", "小熊猫活动场"],
  "queueAndReservation": ["7:30开园即入", "花花别墅排队长，可选择其他别墅"],
  "preparation": ["儿童身份证或户口本", "防晒", "驱蚊", "水"],
  "familyNotes": ["园区较大，可使用观光车"],
  "commonComplaints": ["周末观光车排队长", "下午熊猫常睡觉"],

## Hotel Portfolio Shape

```json
{
  "stayStrategy": "优先住春熙路/天府广场一带，方便地铁、餐饮和夜间补给；预算敏感可退到地铁2/3号线沿线。",
  "source": "Web-Rooter cited + FlyAI booking reference，2026-05-09",
  "options": [
    {
      "tier": "舒适/亲子优先",
      "name": "示例酒店或区域",
      "area": "春熙路/太古里",
      "priceReference": "代表性日期样例价，价格波动",
      "roomType": "FlyAI未返回具体房型，需点击飞猪链接确认",
      "bookingUrl": "https://a.feizhu.com/example",
      "longitude": "104.080455",
      "latitude": "30.66717",
      "fit": "适合带孩子、晚餐选择多、地铁方便",
      "tradeoffs": ["核心区价格更高", "热门日期需提前确认房型"],
      "sourceFactIds": ["fact_chengdu_hotel_area_20260509"]
    }
  ]
}
```

Do not reduce hotels to a cheapest-first list. Include economy, comfortable/family, premium/location-first, or special-stay options when evidence supports those tiers. The renderer accepts either `hotels: [{...}]` or `hotels: { stayStrategy, source, options: [{...}] }`. Specific hotel candidates must include `bookingUrl`/`detailUrl`/`jumpUrl`, a real FlyAI price reference, and a real room type if FlyAI returned one. If FlyAI did not return room type, store the absence explicitly; do not invent room names.

Hotel candidate names must come from `reputation.md` or `itinerary.md` accepted hotel candidate tables. `map-data.json` may fill image/coordinate/link gaps by name match, but it must not add extra hotel options to the rendered portfolio.

If a hotel candidate is accepted into `guidebook-data.json`, preserve FlyAI `longitude`/`latitude` and `mainPic`/`image` when returned, or store geocoded coordinates and image fields from `map-data.json`. If coordinates are unavailable, the renderer will not claim the hotel appears on the map.

## Pruning and Consistency Rules

The guidebook should use the curated Markdown reports as the content authority, then place their facts into the existing dashboard modules. It should preserve useful information without repeating low-value generated summaries or opening report dumps in the main page.

- The Markdown reports should carry the complete decision facts: overall plan, day-by-day route, POI reputation, hotels, budget, food, checklist, and inline freshness notes.
- Structured modules should carry those facts in map interaction, day filtering, drawers, hotel links, and quick scanning surfaces; they are the main traveler-facing guidebook experience.
- Render the original Markdown reports only as collapsed source archive panels so a traveler can verify details without the guidebook becoming a raw report list.
- Keep optional provenance/freshness labels short; internal `fact_*` IDs belong in collapsed technical details.
- Do not duplicate the same warning in every module unless it directly affects that module.
- Budget must be internally consistent with the accepted lodging tier. A `¥498/晚` hotel portfolio cannot share a `3晚 ¥510` accommodation budget unless explicitly labeled as a different budget tier.
- Route summaries should prefer itinerary transport segments. Long Amap walking routes between far-apart POIs are misleading and should be filtered or replaced by driving/transit/taxi references.
- POI experience cards must come from actual attraction/place sections in `reputation.md`. Lodging, hotel candidate, area strategy, traffic, budget, and meta sections are not POIs and must never become “重点景点怎么玩” cards.
- Guidebook validation should fail rather than silently downgrade when Markdown-derived hotels, day activities, daily notes, transport codes/links, or budget rows are missing from `guidebook-data.json`/`guidebook.html`.

## Density Rule

Use adaptive density:

- `brief`: simple city breaks, few warnings, low reservation friction.
- `standard`: normal multi-day city travel.
- `detailed`: family trips, older adults, complex reservations, outdoor risk, high queue friction, or many Amap/FlyAI facts.

The renderer keeps a stable visual style while sections appear only when the corresponding data exists.

## Travel Atlas Interaction Rule

The final `guidebook.html` is a dashboard-style Travel Atlas. Each full-package guidebook should preserve enough structured data for:

- day tabs and timeline cards;
- map marker matching by POI/hotel name;
- itinerary-derived map markers for detailed timeline stops when coordinates are available;
- click-to-open detail drawer;
- expandable POI dossiers with the richest useful reputation/itinerary facts;
- hotel tier cards with Feizhu links, displayed price, room-type status, fit, and tradeoffs;
- inline source notes for volatile or experience-based judgments, with detailed references preserved in `research-ledger.json` and collapsed Markdown archive panels.

Do not reduce `reputation.md`, `itinerary.md`, or `destination-brief.md` to a generic one-paragraph summary if they contain actionable route, queue, reservation, hotel, food, weather, or avoid notes. Convert those facts into structured fields.
