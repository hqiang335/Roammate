# Guidebook Maker

Use this nested skill when the user wants the final interactive travel atlas, printable guidebook, HTML page, or polished artifact for family and friends.

## Contract

Consumes:

- `destination-brief.md`, `reputation.md`, `itinerary.md`, `itinerary-structured.json`, `map-data.json`, and optional `research-ledger.json`.

Produces:

- `guidebook-data.json` generated by `scripts/build-guidebook-data.mjs`.
- `guidebook.html` generated by `scripts/build-guidebook.mjs`.

Does not:

- Research new trip facts by default.
- Hand-write `guidebook-data.json` or `guidebook.html`.
- Summarize `itinerary.md` into lossy cards; it must preserve raw itinerary rows/details via `itinerary-structured.json` and collapsed source archives.
- Introduce traveler-facing hotels or POIs that were not accepted upstream.

Exit gate:

- `node .claude/skills/guidebook-maker/scripts/validate-guidebook.mjs "$TRIP_DIR/guidebook-data.json" "$TRIP_DIR/guidebook.html"` passes.
- Run `qa-guidebook.mjs` when Playwright is available.

## Workflow

1. Gather available inputs in one trip directory: `map-data.json`, `destination-brief.md`, `reputation.md`, `itinerary.md`, `itinerary-structured.json`, and optional `research-ledger.json`. Do not require or read `itinerary-data.json` for new runs.
2. Read `references/guidebook-data-schema.md` only when changing the data contract. Read `references/html-style.md` or `references/guidebook-template.md` only when changing renderer layout or visual behavior. If a rare guidebook-critical external query is truly needed, read `../roammate-travel-concierge/references/tool-command-contract.md` before calling FlyAI or Web-Rooter.
3. Treat `destination-brief.md`, `itinerary.md`, and `reputation.md` as the content authority. Use `itinerary-structured.json` as the lossless machine index of the itinerary, not as a replacement for the Markdown authority. Parse useful facts into the existing guidebook UI modules: overview, daily timeline cards, POI dossiers, hotel cards, food/warnings, budget, checklist, map/drawer notes, and folded source archive.
   - `map-data.json` and `research-ledger.json` are enrichment layers only: coordinates, photos, route estimates, IDs, freshness/source links, and already accepted evidence. They must not replace Markdown hotel candidates, POI experience text, daily itinerary rows, traffic references, or budget rows from the three Markdown reports.
   - If Markdown extraction fails for hotels, POI dossiers, budget, or transport, fix the extractor or upstream Markdown format. Do not silently fall back to unrelated `map-data.json` hotel/POI lists as traveler-facing content.
4. Preserve freshness and uncertainty where they affect traveler decisions: weather, routes, hotels, flights/trains, tickets/packages, semantic tips, and real-experience judgments.
5. Convert accepted FlyAI `ai-search` and Quark/Web-Rooter ledger facts into practical guidebook sections: must-do projects, realistic stay duration, queue strategy, hotel area/tier tradeoffs, restaurant choices, reservation notes, preparation checklist, child/older-adult cautions, rest/meal notes, and skip/avoid choices.
6. Create `guidebook-data.json` with `scripts/build-guidebook-data.mjs`; do not hand-write it for full-package runs. The script should move exact itinerary rows and daily detail fields from `itinerary-structured.json`, then enrich with the three Markdown reports without lossy rewriting.
   - If `guidebook-data.json` only contains thin day activities, the renderer will enrich them from `itinerary.md`; do not strip transport mode, duration, cost, flight/train numbers, booking links, route notes, or hotel links from the handoff.
   - Include `transport.flights`, `transport.trains`, and `transport.local` whenever available. Preserve `url`/`bookingUrl`/`ticketUrl` from FlyAI, 12306, airline, or Feizhu references so the "交通参考" cards can show a "订票/查询" link. If these facts only exist in `itinerary.md`, keep the markdown links intact so the renderer can parse them.
   - Keep transport rows visually grouped as 去程、返程、市内交通 or 城际备选; do not mix all movement references into one undifferentiated card grid.
   - Include `food` when you have structured dining recommendations. If food only appears in `destination-brief.md`, `itinerary.md`, or `reputation.md`, do not omit those files; the renderer will derive dining suggestions and warnings from meal rows, food bullets, and "不要吃正餐/小吃偏贵" reputation notes.
   - POI dossier parsing must only use actual attraction/place sections from `reputation.md`. Exclude lodging/meta sections such as "酒店区域与候选", "住宿区域", "候选酒店", "区域判断", "餐厅推荐", and budget/traffic sections.
   - Hotel cards must come from `reputation.md`/`itinerary.md` accepted hotel candidate tables. The renderer may attach photos, coordinates, and booking links from `map-data.json` by name match, but it must not introduce new hotel names that are absent from the Markdown candidates.
7. Do not rework the ledger just for guidebook rendering. Preserve fact IDs already present when useful, then let the generator script build the compact handoff.
8. Generate `guidebook.html` with `scripts/build-guidebook.mjs`. The renderer produces a dashboard-style Travel Atlas, keeps the original UI skeleton, uses sibling `map-data.json` for coordinates/routes/photos, and renders the three Markdown reports only as collapsed source archive panels for audit. It intentionally ignores `itinerary-data.json` if an old file exists in the directory.
9. Validate `guidebook.html` with `scripts/validate-guidebook.mjs`.
   - Treat validation failures about missing Markdown hotel names, daily activity rows, daily detail snippets, transport codes/URLs, budget labels, or lodging sections appearing as POI cards as hard blockers. These are signs that the Markdown-to-guidebook migration is lossy or polluted by an intermediate JSON fallback.
   - If validation says `guidebook-data.json` is missing a hotel/POI already present in `reputation.md` or `itinerary.md`, fix the parser, merge logic, or accepted-candidate handoff. Do not edit the upstream Markdown merely to make validation pass unless the Markdown fact itself is wrong.
10. Run `scripts/qa-guidebook.mjs` when Playwright is available to check desktop/mobile rendering, drawer navigation, and horizontal overflow.

## Research Budget

Guidebook Maker is a rendering and migration stage. It should not perform broad research.

- Default: run zero new Amap/FlyAI/Web-Rooter queries.
- If validation reveals a guidebook-critical missing fact, first fix Markdown parsing or upstream Markdown structure.
- Only query externally when the three Markdown files and `map-data.json` truly lack a required coordinate/photo/link needed for rendering.
- Never print raw JSON or full article text into the conversation; guidebook scripts and validators should provide compact summaries.

## Required Sections

- Cover / atlas header
- Trip overview dashboard
- Compact overall itinerary plan: theme, area, intensity, core experience, key reminder
- Interactive daily itinerary with day tabs
- Map and transport workspace, using `map-data.json` when present
- POI experience dossiers: recommended duration, must-do, queue/reservation, prep, family cautions, avoid notes, source label
- Stay areas and hotel portfolio when lodging choices matter
- Food and reputation warnings
- Tickets, reservations, and booking-market references when available
- Budget
- Pre-trip checklist
- Collapsed source archive panels for sibling `destination-brief.md`, `itinerary.md`, and `reputation.md`; their facts should already appear in the corresponding UI modules

These are base capabilities, not a fixed length requirement. Optional modules should appear only when the corresponding data is useful. For example, long avoidance sections are appropriate only when there are meaningful avoidance signals; complex preparation sections are appropriate only when the trip requires them.

## Rules

- Simplified Chinese by default.
- No AI-generated images in V1.
- Do not require external assets. CDN icons are optional only if there is a text fallback.
- Keep the design dashboard-first, visually polished, printable, and mobile-readable. The default layout is a left itinerary panel plus interactive map workspace, with details revealed by cards, drawers, and expandable dossiers.
- Prices, opening hours, weather, hotel availability, and routes must include freshness or uncertainty notes inside the relevant modules; do not add a standalone "来源与可信度" section to the guidebook.
- Do not present FlyAI `ai-search` advice as official policy; label it as semantic reference.
- Do not present FlyAI hotel inventory as final hotel judgment without Quark/Web-Rooter area or experience context when such evidence exists.
- Every accepted hotel candidate in the guidebook must include a clickable Feizhu link, displayed FlyAI price, tier/type, and real room type if returned. If room type was not returned by FlyAI, say so explicitly instead of inventing a room.
- The lodging section should present hotel cards only; area strategy belongs inside card fields or the folded source archive, not as a loose paragraph above the cards.
- Public web evidence used for decisions should be preserved in `research-ledger.json` or concise inline context; the guidebook itself should not render an empty standalone source module.
- The guidebook should help the traveler decide what to do inside each attraction, not just list attraction names. Do not shrink reputation or itinerary experience into a thin summary when structured facts exist.
- The final guidebook must not be a lossy rewrite of `destination-brief.md`, `itinerary.md`, or `reputation.md`. Preserve the existing dashboard layout and move the reports' facts into matching cards, chips, tables, drawers, tabs, hotels, budget rows, and checklist groups.
- Do not inflate the page with low-value repetition or with open report dumps. The Markdown reports remain collapsed for verification; the main reading path should be the polished interactive guidebook UI populated from those reports.
- Budget and lodging must use one consistent data source. If the hotel portfolio changes nightly price, update accommodation and total budget or clearly mark the budget as a different tier.
- Map route text must prefer itinerary transport segments. Do not present cross-city Amap walking routes as recommended movement; long walking segments should be filtered, relabeled as estimates, or replaced by driving/transit/taxi references.
- Only claim hotel map display when accepted hotel candidates have coordinates.
- Day filters on the map must surface the same meaningful POIs mentioned in the detailed daily timeline, including stations, core plazas/streets, accepted hotels, food stops, and named attractions when coordinates are available.
- Mobile rendering is part of completion: no horizontal scrolling at common phone widths, and drawers must close on navigation or ESC.
- Do not hand-write the final `guidebook.html`.
- Do not create temporary Python, Bash heredoc, or ad hoc scripts to assemble long guidebook HTML.
- If direct `Write` of `guidebook-data.json` fails, reduce or split the JSON data, then run the fixed renderer; do not switch to hand-written HTML, shell heredocs, or ad hoc `cat > guidebook-data.json` assembly.
- If `validate-guidebook.mjs` fails, fix `guidebook-data.json` or the renderer and rerun validation before delivering.
- If validation fails because equivalent source text is escaped differently in JSON or HTML, fix the parser/renderer/validator and regenerate. Do not remove punctuation, quotes, Chinese book-title marks, or other useful source wording from `itinerary.md`, `destination-brief.md`, or `reputation.md` just to make string matching pass.
- The visual style is owned by `scripts/build-guidebook.mjs`; do not invent a new CSS theme per trip. Use stable layout, components, spacing, marker categories, day tabs, drawers, cards, and print behavior across all destinations. Destination personality may affect data and labels, not the page skeleton.
- Do not run new Amap/FlyAI/Web-Rooter searches from guidebook-maker unless the three Markdown reports, `research-ledger.json`, and `map-data.json` lack a needed guidebook-critical fact.
- `guidebook-data.json` should stay compact and may reference ledger fact IDs when useful; it should not duplicate raw search outputs or whole Markdown reports. It should contain the structured facts parsed from the reports so the renderer can fill the stable guidebook UI.
